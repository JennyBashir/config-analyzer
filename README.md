
# Config Analyzer

Config Analyzer — утилита на Go для анализа конфигурационных файлов веб-приложений.

Она поддерживает JSON и YAML, проверяет конфигурацию по набору правил и сообщает о потенциально опасных настройках.

Проект выполнен в рамках тестового задания.

## Что умеет

Утилита проверяет конфигурацию на наличие следующих проблем:

* включён debug-режим логирования;
* пароль или секрет хранится в открытом виде;
* используется адрес `0.0.0.0`;
* отключён TLS;
* отключена проверка TLS-сертификата;
* используются небезопасные алгоритмы (`MD5`, `SHA1`, `DES`, `3DES`).

Для каждой найденной проблемы выводятся:

* уровень опасности (`LOW`, `MEDIUM`, `HIGH`);
* краткое описание;
* рекомендация по исправлению.

Если найдена хотя бы одна проблема, программа завершается с кодом ошибки.

## Требования

* Go 1.20+

## Сборка

Клонируйте репозиторий:

```bash
git clone https://github.com/JennyBashir/config-analyzer.git
cd config-analyzer
```

Установите зависимости:

```bash
go mod download
```

Соберите приложение:

```bash
go build -o config-analyzer ./cmd/main
```

## Использование

Проверить YAML-файл:

```bash
./config-analyzer config.yaml
```

Проверить JSON-файл:

```bash
./config-analyzer config.json
```

Без предварительной сборки:

```bash
go run ./cmd/main config.yaml
```

## Флаги

### `-s`, `--silent`

Не завершать программу с ошибкой, если были найдены проблемы.

Например:

```bash
./config-analyzer --silent config.yaml
```

или

```bash
./config-analyzer -s config.yaml
```

Все найденные проблемы будут показаны, но код завершения будет `0`.

### `--stdin`

Позволяет прочитать конфигурацию из стандартного ввода.

YAML:

```bash
cat config.yaml | ./config-analyzer --stdin
```

JSON:

```bash
echo '{"log":{"level":"debug"}}' | ./config-analyzer --stdin
```

Можно использовать вместе с `--silent`:

```bash
cat config.yaml | ./config-analyzer --stdin --silent
```

## Пример

Конфигурация:

```yaml
log:
  level: debug

server:
  host: 0.0.0.0

tls:
  enabled: false

database:
  password: mypassword

security:
  algorithm: md5
```

Результат:

```text
LOW: Логирование в debug-режиме
Recommendation: Используйте уровень info или выше.

MEDIUM: Приложение привязано к адресу 0.0.0.0
Recommendation: Ограничьте адрес прослушивания.

HIGH: TLS отключен
Recommendation: Включите TLS.
```

## Коды завершения

| Код | Описание                               |
| --: | -------------------------------------- |
|   0 | Проблем не найдено                     |
|   1 | Найдены проблемы                       |
|   1 | Ошибка чтения или разбора конфигурации |

Если используется `-s` или `--silent`, программа завершится с кодом `0`, даже если проблемы были найдены.

## Поддерживаемые форматы

* `.json`
* `.yaml`
* `.yml`

При использовании `--stdin` утилита сама определяет формат входных данных.

## gRPC

В проекте также реализованы gRPC-сервер и пример клиента.

Через него можно отправить конфигурацию на проверку и получить список найденных проблем.

### Запуск сервера

```bash
go run ./cmd/server
```

После запуска сервер слушает:

```text
localhost:50051
```

Для остановки достаточно нажать `Ctrl+C`.

### Запуск клиента

В отдельном терминале:

```bash
go run ./cmd/client
```

Клиент подключается к серверу, отправляет конфигурацию и выводит найденные проблемы.

### gRPC API

Описание сервиса находится в файле:

```text
proto/analyzer.proto
```

Основной метод:

```protobuf
rpc Analyze(AnalyzeRequest) returns (AnalyzeResponse);
```

Запрос содержит конфигурацию в виде строки, а ответ — список найденных проблем.

## Описание пакетов

### `cmd`

Точки входа приложения:

* консольная утилита;
* gRPC-сервер;
* пример gRPC-клиента.

### `internal/cli`

Обрабатывает аргументы командной строки и загружает конфигурацию из файла или `stdin`.

### `internal/config`

Отвечает за чтение и парсинг JSON и YAML.

### `internal/walker`

Рекурсивно обходит всю конфигурацию и передаёт найденные значения правилам проверки.

### `internal/rules`

Содержит отдельные проверки безопасности.

Каждое правило находится в собственном файле.

### `internal/analyzer`

Запускает все правила и собирает найденные проблемы в один список.

### `internal/grpc`

Реализация gRPC-сервиса.

### `internal/types`

Общие структуры данных, используемые в проекте.

### `proto`

Описание gRPC API.

## Как добавить новое правило

Структура проекта позволяет достаточно просто добавить новую проверку.

Для этого нужно:

1. создать новое правило в `internal/rules`;
2. использовать `walker.Walk()` для обхода конфигурации;
3. вернуть найденные проблемы как `types.Issue`;
4. добавить вызов нового правила в `analyzer.Analyze`;
5. написать тесты.
